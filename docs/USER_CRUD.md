# User CRUD — Endpoint Design

**Router:** `backend/app/routers/auth.py`
**Service:** `backend/app/services/auth.py`
**Prefix:** `/auth`

Design philosophy: **passwordless, victim-first, deletable**. No password database to breach. No tracking cookies. Delete means delete.

## The Magic-Link Flow

Three-step flow. No passwords.

```
[1] POST /auth/login  {email}
         └─> system generates MagicLink (token, 24h TTL)
         └─> returns token (MVP) / emails it (production)

[2] POST /auth/verify  {token}
         └─> validates token (not used, not expired)
         └─> creates Session (30-day TTL)
         └─> returns session_token

[3] Any authenticated request:
         Authorization: Bearer <session_token>
         └─> get_user_by_session(token) → User
```

**Why magic links:**
- No password = no password reuse risk, no breach exposure
- Email ownership = identity (same model as Slack, Medium, etc.)
- Tokens are single-use + expiring = phishing-resistant
- No persistent cookies needed → better for victims on shared devices

**Trade-offs:**
- Requires email deliverability (won't work if the email provider blocks us)
- User must check email every ~30 days
- SMS as fallback not yet implemented

## Endpoints

### `POST /auth/login` — Request magic link

**Request:**
```json
{"email": "victim@example.com"}
```

**Response:**
```json
{
  "message": "Magic link sent to victim@example.com. Check your inbox.",
  "magic_link_token": "Abc123..."   // MVP only, remove in production
}
```

**Key design points:**
- Email is lowercased before lookup → no case-sensitivity issues
- If the email doesn't exist, we still return a success message (avoid email enumeration attacks)
- In MVP, the token is returned directly for testing. Production: only sent via email, response contains just the message.

### `POST /auth/verify` — Exchange magic link for session

**Request:**
```json
{"token": "Abc123..."}
```

**Response:**
```json
{
  "session_token": "Xyz789...",
  "expires_at": "2026-05-12T14:23:00Z",
  "user": {
    "id": "uuid",
    "email": "victim@example.com",
    "display_name": null,
    "lang": "de"
  }
}
```

**Key design points:**
- Magic link token is marked `used=True` immediately to prevent replay
- Session token is 256-bit random (`secrets.token_urlsafe(32)`)
- Session TTL: 30 days (configurable via env)

### `GET /auth/me` — Current user

Requires `Authorization: Bearer <session_token>`. Returns full user record.

### `PUT /auth/me` — Update profile

```json
{"display_name": "Jana", "lang": "de"}
```

Only non-null fields are updated. Language is validated against `{"de", "en"}`.

### `POST /auth/logout` — Invalidate session

Marks the current session's `active=False`. User must re-login via magic link.

### `DELETE /auth/me` — Soft delete (GDPR)

**Trigger:** user clicks "delete my account".

**Behavior:**
- `users.deleted_at` set to `now()`
- User data hidden from all reads immediately (queries filter `deleted_at IS NULL`)
- Session invalidated
- **Recovery window: 7 days.** If user logs in within 7 days, `deleted_at` is cleared.
- After 7 days: hard-delete job removes all rows (user, cases, evidence, classifications).

**Response:**
```json
{
  "message": "Account scheduled for deletion. Data will be permanently removed in 7 days.",
  "hard_delete_after": "7 days"
}
```

### `DELETE /auth/me/emergency` — Immediate hard delete

**Trigger:** user in danger, needs everything gone NOW (e.g., abuser approaching the device).

**Behavior:**
- All user data deleted synchronously and irreversibly
- No recovery window
- No confirmation email (that would defeat the purpose)

**Response:**
```json
{
  "message": "All data permanently deleted. This cannot be undone.",
  "recovered": false
}
```

**Why emergency delete exists:** real victims have reported apps that made "delete" go through 3 confirmations and a 24h cooldown. For someone whose abuser is in the next room, that's unsafe. Emergency delete is the escape hatch.

## Session Management

**Session storage:** `users` + `sessions` tables (currently implemented in memory in `services/auth.py`, planned SQLAlchemy migration in Sprint 1).

**Session token:** 256-bit random, generated by `secrets.token_urlsafe(32)`.

**Expiry:** 30 days from creation. On each authenticated request, we check `expires_at > now()` and `active=True`.

**Logout:** sets `active=False`. Does NOT delete the row — useful for session audit logs.

## CRUD Coverage

| Operation | Endpoint | Method |
|-----------|----------|--------|
| Create | `POST /auth/login` + `POST /auth/verify` | Two-step |
| Read | `GET /auth/me` | Single user only (no admin list in MVP) |
| Update | `PUT /auth/me` | display_name, lang |
| Delete (soft) | `DELETE /auth/me` | 7-day recovery |
| Delete (hard) | `DELETE /auth/me/emergency` | Immediate |

**Missing by design in MVP:**
- No admin endpoint to list/search users (privacy-first)
- No password change (no passwords)
- No email change (email IS the identity; user must create new account)

**Next milestone (Sprint 1):**
- Organization accounts (multi-tenant): a user can belong to an NGO/org, case ownership extends to org
- Role-based access (org admin, case worker, viewer)

## Security Invariants

1. **Email enumeration protected** — `/login` doesn't confirm whether the email exists.
2. **Magic link single-use** — `used=True` set atomically on `/verify`.
3. **Session expiry enforced on every request** — not just on creation.
4. **Logout invalidates server-side** — not just client cookie-clearing.
5. **Emergency delete is unrecoverable** — by design. There's no "oops" escape.
6. **Rate limiting** (Sprint 1 hardening): `/login` limited to 5 requests per IP per hour, preventing magic-link spam.

## Error Handling

| HTTP | When |
|------|------|
| 400 | Invalid email format |
| 401 | Missing / invalid / expired session token |
| 404 | (never used — magic link errors are 401 to prevent enumeration) |
| 429 | (Sprint 1) too many login attempts |

## Open Questions for Tutor

1. **Magic link in URL is visible in browser history.** Is this an acceptable risk? Trade-off vs sending a 6-digit OTP.
2. **Soft-delete is 7 days.** Is that too long (attacker gets 7d window to threaten recovery) or too short (user regret)?
3. **Emergency delete requires no auth token check beyond the existing session.** Should we require a second factor? Trade-off: safety vs friction for someone in immediate danger.
