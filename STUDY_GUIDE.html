How to build SafeVoice — from an empty folder.

If I had to rebuild this tomorrow, in order, this is what I would do. Each chapter follows one shape: why (the design decision), what (what to build), how (the code, step by step), and gotchas (mistakes I already made, so you don't have to).

01 · Setup — FastAPI + SQLAlchemy skeleton.

Why FastAPI gives you automatic OpenAPI docs at /docs, Pydantic validation for free, and async if you ever need it. SQLAlchemy gives you a real ORM that works with both SQLite (local, zero setup) and Postgres (production). Pick these two and you can move fast without painting yourself into a corner.

Steps

Create the project folder and a virtual environment. 
mkdir safevoice && cd safevoice
mkdir backend && cd backend
python3 -m venv venv
source venv/bin/activate
Install the core dependencies. 
pip install fastapi uvicorn sqlalchemy alembic openai pydantic python-dotenv
pip freeze > requirements.txt
Create the app skeleton. Folder structure: 
backend/
├── app/
│   ├── __init__.py
│   ├── main.py              # FastAPI entry point
│   ├── database.py          # SQLAlchemy setup + models + seed
│   ├── routers/             # one file per resource (auth, cases, analyze)
│   │   └── __init__.py
│   └── services/            # business logic (classifier, legal_ai)
│       └── __init__.py
├── .env                     # OPENAI_API_KEY (NEVER committed)
├── .gitignore               # venv/, .env, __pycache__/
└── requirements.txt
Minimal main.py that wires routers. 
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware
from app.routers import auth, cases, analyze
from app.database import init_db, seed_categories_and_laws

app = FastAPI(title="SafeVoice API", version="1.0")
app.add_middleware(CORSMiddleware, allow_origins=["*"], allow_methods=["*"])

@app.on_event("startup")
def on_startup():
    init_db()
    seed_categories_and_laws()

app.include_router(auth.router,    prefix="/auth",    tags=["auth"])
app.include_router(cases.router,   prefix="/cases",   tags=["cases"])
app.include_router(analyze.router, prefix="/analyze", tags=["analyze"])

@app.get("/health")
def health():
    return {"ok": True}
Run it. 
uvicorn app.main:app --reload
# → http://localhost:8000/docs
Gotcha Never commit .env. Add it to .gitignore before your first commit. If you do commit a key by accident, rotate it immediately — scanning bots index public repos within minutes.

02 · Database — label every column.

Why The first question any mentor will ask about the schema is "which columns did the user fill in, and which did the AI?" — and it is the right question. If you can't answer it in one glance, your schema is not trustworthy. The fix is a convention: every column is either user input, AI populated, or system generated. Nothing else. The labels live as inline comments in database.py and as note: entries in schema.dbml.

Steps

Sketch the six tables on paper before writing a line of SQL. 
users            — who was harassed?
evidence_items   — what exactly happened?
cases            — which incidents belong together?
classifications  — what did the AI think?
categories       — what counts as "harassment"?
laws             — what counts as "§ 241 StGB"?
+ two M:N junction tables (classification_categories, classification_laws)
Write database.py with one model per table. Use a comment on every column. 
class Classification(Base):
    __tablename__ = "classifications"

    id                = Column(String, primary_key=True, default=gen_uuid)         # system_generated
    evidence_item_id  = Column(String, ForeignKey("evidence_items.id"), unique=True)  # 1-to-1 link
    severity          = Column(String, default="none")       # AI_POPULATED
    confidence        = Column(Float,  default=0.0)          # AI_POPULATED 0-1
    summary           = Column(Text)                         # AI_POPULATED (en)
    summary_de        = Column(Text)                         # AI_POPULATED (de)
    classified_at     = Column(DateTime, default=datetime.utcnow)  # system_generated
Add junction tables for the many-to-many relationships. 
classification_categories = Table(
    "classification_categories", Base.metadata,
    Column("classification_id", String, ForeignKey("classifications.id"), primary_key=True),
    Column("category_id",       String, ForeignKey("categories.id"),       primary_key=True),
)
Seed reference data on startup, so a fresh DB ships with categories and laws already in it. 
def seed_categories_and_laws():
    db = SessionLocal()
    for id_name, name, name_de in CATEGORY_SEEDS:
        if not db.query(Category).filter_by(id=id_name).first():
            db.add(Category(id=id_name, name=name, name_de=name_de))
    for id_name, code, section, name, name_de, penalty in LAW_SEEDS:
        if not db.query(Law).filter_by(id=id_name).first():
            db.add(Law(id=id_name, code=code, section=section,
                       name=name, name_de=name_de, max_penalty=penalty))
    db.commit(); db.close()
Export a DBML mirror at schema.dbml for dbdiagram.io. 
Table classifications {
  id varchar [pk]
  severity varchar [note: 'AI POPULATED']
  confidence float [note: 'AI POPULATED — 0.0-1.0']
  summary text [note: 'AI POPULATED — English']
  ...
}
Gotcha Don't store categories or laws as a comma-separated string on classifications. You'll regret it the first time you try "show me every case citing § 241 StGB" — a JOIN on a junction table is one query; a LIKE on a string is a nightmare.

03 · User CRUD — magic link, zero passwords.

Why A victim under stress forgets passwords. A password database we don't have cannot leak. Magic link is a 15-minute single-use token sent by email — log in by clicking a link, no password round-trip. Safer, kinder, and simpler to build.

Steps

Create routers/auth.py. Plan 8 endpoints upfront — you will thank yourself later. 
# routers/auth.py
POST   /auth/login           # request magic link (creates user on first login)
POST   /auth/verify          # exchange token for session
GET    /auth/me              # read current user
PUT    /auth/me              # update display_name, language
POST   /auth/logout          # end session
DELETE /auth/me              # soft delete (GDPR Art. 17)
DELETE /auth/me/emergency    # hard delete (purge cases + evidence)
GET    /auth/me/export       # Art. 20 data export
Implement /auth/login — create user on first touch, issue a token. 
@router.post("/login")
def login(req: LoginRequest, db: Session = Depends(get_db)):
    user = db.query(User).filter_by(email=req.email).first()
    if not user:
        user = User(email=req.email, language=req.language or "de")
        db.add(user); db.commit(); db.refresh(user)

    token = secrets.token_urlsafe(32)       # 256 bits of entropy
    expires = datetime.utcnow() + timedelta(minutes=15)
    db.add(MagicLinkToken(user_id=user.id, token=token, expires_at=expires))
    db.commit()

    # MVP: return the token directly. Production: email it via Resend.
    return {"ok": True, "token": token}
Implement /auth/verify — one-shot exchange, token → session. 
@router.post("/verify")
def verify(req: VerifyRequest, db: Session = Depends(get_db)):
    row = db.query(MagicLinkToken).filter_by(token=req.token).first()
    if not row or row.used_at or row.expires_at < datetime.utcnow():
        raise HTTPException(401, "invalid_or_expired")

    row.used_at = datetime.utcnow()          # single-use
    session = Session_(
        user_id=row.user_id,
        token=secrets.token_urlsafe(32),
        expires_at=datetime.utcnow() + timedelta(days=30),
    )
    db.add(session); db.commit()
    return {"session_token": session.token, "user": user_to_dict(row.user)}
Write a current_user FastAPI dependency. Every protected route uses it. 
def current_user(authorization: str = Header(...), db: Session = Depends(get_db)):
    token = authorization.replace("Bearer ", "")
    session = db.query(Session_).filter_by(token=token).first()
    if not session or session.expires_at < datetime.utcnow():
        raise HTTPException(401, "invalid_session")
    return session.user
Add soft and hard delete for GDPR. 
@router.delete("/me")        # Art. 17 — soft delete, grace period
def soft_delete(user = Depends(current_user), db = Depends(get_db)):
    user.deleted_at = datetime.utcnow()
    db.commit(); return {"ok": True, "grace_period_days": 7}

@router.delete("/me/emergency")   # immediate purge — victim under threat
def hard_delete(user = Depends(current_user), db = Depends(get_db)):
    db.query(Case).filter_by(user_id=user.id).delete()
    db.query(User).filter_by(id=user.id).delete()
    db.commit(); return {"ok": True}
Gotcha Make the session token a separate value from the magic-link token. Magic-link tokens are short-lived (15 min) and single-use; session tokens are long-lived (30 days) and reusable. Mixing them means a stolen magic link gives the attacker 30 days of access.

04 · Case CRUD — never created by the user.

Why A harassment victim is already doing enough emotional work. Asking them to also think "do I create a new case, or add to an existing one?" is a cliff between pain and help. So: we expose POST /cases/ for completeness, but the real flow creates cases implicitly inside POST /analyze/ingest. One paste, one transaction.

Steps

Implement the standard six CRUD endpoints in routers/cases.py. 
GET    /cases/               # list (current user's cases)
GET    /cases/{id}           # detail + evidence + classifications
POST   /cases/               # explicit create — rarely used directly
PUT    /cases/{id}           # update title, status, notes
DELETE /cases/{id}           # delete (cascade to evidence + classifications)
POST   /cases/{id}/evidence  # "filling" — attach evidence to an existing case
Implement the real entry point in routers/analyze.py — the one the UI calls. 
@router.post("/ingest")
def ingest(req: IngestRequest, user = Depends(current_user), db = Depends(get_db)):
    # 1. create the evidence record first (hashed + timestamped)
    ev = EvidenceItem(
        raw_content=req.content,
        content_hash=sha256(req.content.encode()).hexdigest(),
        hash_chain_previous=_latest_hash_for(user.id, db),
        timestamp_utc=datetime.utcnow(),
    )

    # 2. classify the content (one LLM call — see chapter 05)
    result = classifier.classify(req.content, victim_context=req.victim_context)

    # 3. create the case FROM the classification (not before)
    case = Case(user_id=user.id,
                title=result.summary[:80],
                overall_severity=result.severity,
                status="open")
    db.add(case); db.flush()             # get the case.id
    ev.case_id = case.id
    db.add(ev)
    db.add(Classification(evidence_item_id=ev.id, **result.as_dict()))
    db.commit()
    return {"case_id": case.id, "severity": result.severity}
When the user comes back with a related comment, route to /cases/{id}/evidence instead — same classification pipeline, but attach to the existing case.
Gotcha The order in ingest matters. Hash the evidence BEFORE you classify it, so the hash covers the exact text that entered the system — not whatever comes back from the classifier. The hash chain's whole point is: if anyone tampers with past evidence, the next verify breaks.

05 · AI flow — input, context, system prompt.

Why This is the action item with three exclamation marks. Get it wrong and the whole app is guessing; get it right and the whole app has a spine. Three questions to answer, in this order: what is the input?, what is the context?, what is the system prompt?

Answers

1 · Input — always a string

Text arrives three ways: paste, URL scrape (served by services/scraper.py), screenshot OCR (served by services/upload.py). By the time classify() runs, it's plain UTF-8 text. One format, one path. If you build three classifiers for three inputs, you'll debug three bugs.

2 · Context — lives in the system prompt

No RAG at this layer. The classifier's context is the stable list of categories, laws, severity definitions, and the ethical rule — "when in doubt, err on the side of the victim." Because these rules don't change per request, they belong in the system prompt, not retrieved documents.

3 · System prompt — few-shot, with victim_context handling

The prompt has five blocks. Keep each one short and load-bearing.

Du bist SafeVoice — Klassifikator für digitale Gewalt in Deutschland.

GRUNDREGELN
- Verstehe Tippfehler, Slang, Verschleierung (f0tze, stirbt statt stirb, H*re, 1488)
- Bei Mehrdeutigkeit: im Zweifel FÜR das Opfer (höhere Severity)
- Eine Drohung ist eine Drohung, auch indirekt ("Ich weiß wo du wohnst")
- Redewendungen sind keine Straftaten ("Das bringt mich um" = Idiom, severity=none)
- NetzDG § 3 gilt IMMER bei Social Media Inhalten

VICTIM_CONTEXT VERWENDEN (wenn vorhanden)
- Ex-Partner → § 238 StGB (Stalking), nicht nur § 241 StGB
- Arbeitgeber / Kollege → § 185 StGB wiegt schwerer (Druckverhältnis)
- Minderjährig → bei sexuellem Inhalt mögliches § 184b/h StGB

SEVERITY (mit Beispielen)
- low:    "Bist du dumm?" — Grenzwertig
- medium: "Du H*re" — § 185 StGB
- high:   "Ich weiß wo du wohnst" — § 241 StGB
- critical: "Ich bringe dich um" — § 241 + § 126a StGB

BEISPIELE (few-shot)
Input: "Ich bringe dich um, du Drecksschlampe"
  → severity=critical, categories=[death_threat, misogyny]
Input: "Das bringt mich um" (Idiom)
  → severity=low, categories=[harassment]
Input: "Stirbt endlich, du H*re" (Obfuscation)
  → severity=critical, categories=[death_threat, misogyny]

SUMMARY-QUALITÄT
- 1–2 faktische Sätze, kein Drama, keine Wertung
Why few-shot, not zero-shot? Zero-shot gets obfuscation wrong ~15% of the time and flags German idioms as threats ~20% of the time. Adding four worked examples — especially a false-positive idiom and an obfuscation case — removes both failure modes. The cost is ~300 extra tokens per call (~€0.00005 on gpt-4o-mini).
Gotcha Keep the system prompt in a constant at the top of your classifier module (SYSTEM_PROMPT = """..."""), not buried inside a function. You will want to diff it when you re-tune. If it ends up as an f-string with dynamic user fields, you have accidentally built a prompt-injection hole — split user input into the user message, never into the system.

06 · Classification endpoints — four entries, one brain.

Why Different inputs (free text, URL, screenshot) and different intents (classify-only vs persist) need different HTTP endpoints — but they all call the same classifier function. Keep the classifier stateless and pure; put side-effects in the router. This keeps unit tests fast and one-bug-one-place honest.

Steps

Declare the four entry points in routers/analyze.py. 
POST /analyze/text    — classify a string, NO DB write (fastest path)
POST /analyze/ingest  — classify + persist evidence + create case
POST /analyze/url     — scrape URL → classify
POST /analyze/case    — case-level RAG across N evidence items (chapter 08)
Each route normalises its input to a string, then calls the classifier. 
@router.post("/text")
def classify_text(req: TextRequest):
    result = classifier.classify(req.content, victim_context=req.victim_context)
    return result.as_dict()            # pure — no DB

@router.post("/url")
def classify_url(req: UrlRequest):
    text = scraper.scrape(req.url)     # → string
    return classifier.classify(text).as_dict()
The classifier itself lives in services/classifier.py — one public function, stateless. 
def classify(text: str, *, victim_context: str | None = None) -> ClassificationResult:
    result = classifier_llm_v2.classify_with_llm(text, victim_context=victim_context)
    if result is None:
        raise ClassifierUnavailableError()   # → 503 in the router
    return result
Gotcha Resist the urge to do "one endpoint, a union type input." You'll end up with a Pydantic model with five optional fields and a router that branches on if req.url:... elif req.text: .... Four small endpoints is cleaner than one fat one.

07 · Structured Outputs — .parse() + Pydantic.

Why The older pattern was: ask the model for JSON, hope it returns JSON, json.loads, catch five kinds of exceptions, regex-strip markdown code fences, retry. OpenAI's Structured Outputs eliminates all of that. You give the API a Pydantic schema; the API validates the response server-side; you either get a typed Pydantic instance, or you get None. One call, one branch, zero defensive parsing.

Steps

Define the shape of the answer as a Pydantic model. 
from pydantic import BaseModel, Field, ConfigDict
from enum import Enum

class LLMSeverity(str, Enum):
    low = "low"; medium = "medium"; high = "high"; critical = "critical"

class LLMCategory(str, Enum):
    harassment = "harassment"
    threat     = "threat"
    misogyny   = "misogyny"
    # ... 15 total, exhaustive

class LLMLaw(str, Enum):
    stgb_185  = "§ 185 StGB"
    stgb_241  = "§ 241 StGB"
    netzdg_3  = "NetzDG § 3"
    # ... 11 total, exhaustive

class LLMClassification(BaseModel):
    model_config = ConfigDict(extra="forbid")        # reject unknown keys

    severity:                 LLMSeverity
    categories:               list[LLMCategory] = Field(..., min_length=1)
    confidence:               float = Field(..., ge=0.0, le=1.0)
    requires_immediate_action: bool
    summary:                  str
    summary_de:               str
    applicable_laws:          list[LLMLaw]
    potential_consequences:    str
    potential_consequences_de: str
Call client.chat.completions.parse(), not .create(). 
from openai import OpenAI
client = OpenAI()

completion = client.chat.completions.parse(
    model="gpt-4o-mini",
    temperature=0,                     # legal work: deterministic
    max_tokens=1024,
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user",   "content": f"Klassifiziere diesen Inhalt:\n\n{text}"},
    ],
    response_format=LLMClassification, # ← Pydantic, server-enforced
)
Handle the two failure modes — refusal and schema failure — explicitly. 
msg = completion.choices[0].message

if msg.refusal:
    logger.warning(f"LLM refused: {msg.refusal}")
    return None                        # clean 503 from orchestrator

parsed = msg.parsed                    # Pydantic instance OR None
if parsed is None:
    return None                        # same — clean 503

return _to_domain(parsed)              # map enums to internal types
Gotcha Use enums, not plain strings, for categories and laws. A plain str field would let the model invent a new category ("hate-speech-ish-maybe") and Pydantic would accept it. An Enum with exhaustive members makes the API refuse the response before it ever reaches your code.
Gotcha 2 Structured Outputs requires every field to be required. If a field is optional, model it as MyField | None — do not leave it out of required.

08 · RAG at case level — where retrieval earns its place.

Why The classifier doesn't need RAG — it takes one string and returns one verdict. But when the user asks "what does my whole case look like?" — that's reasoning across N evidence items at once. That's exactly what retrieve → augment → generate is for.

Steps

Create services/legal_ai.py. The function takes a case ID, returns a case-level analysis. 
def analyze_case_legally(case_id: str, db: Session) -> CaseAnalysis:
    # 1. RETRIEVE — pull evidence + classifications from the relational DB
    case = db.query(Case).filter_by(id=case_id).first()
    evidence = db.query(EvidenceItem).filter_by(case_id=case_id).all()

    # 2. AUGMENT — build a context block for the prompt
    context = _format_evidence_context(case, evidence)

    # 3. GENERATE — structured output, same .parse() pattern
    completion = client.chat.completions.parse(
        model="gpt-4o-mini",
        messages=[
            {"role": "system", "content": CASE_ANALYSIS_PROMPT},
            {"role": "user",   "content": context},
        ],
        response_format=CaseAnalysis,
    )
    return completion.choices[0].message.parsed
Expose at POST /analyze/case.
Design note You don't need a vector database to call this RAG. The "retrieval" here is a plain SQL query against the relational DB. The pattern is retrieve-then-generate; the infrastructure is whatever finds the relevant data. A vector DB is one implementation of retrieval, not the definition of RAG.
