action item 1

Database schema — user input vs AI-populated vs system-generated

Zisis asked: for each table, which attributes are user input, which are AI-populated, which are system-generated?

Eight tables, real SQLAlchemy ORM with Alembic migrations. SQLite locally (safevoice.db), Postgres in production. Every column has a provenance:

  • User input — email, language, content, victim_context
  • System-generated — IDs, hashes, UTC timestamps, platform auto-detection
  • AI-populated — severity, categories, laws, summaries, consequences
backend/app/database.py 
class User(Base):
    __tablename__ = "users"

    id           = Column(String, primary_key=True, default=gen_uuid)   # system
    email        = Column(String, unique=True, nullable=False)          # user input
    display_name = Column(String)                                       # user input
    language     = Column(String, default="de")                         # user input
    created_at   = Column(DateTime, default=datetime.utcnow)            # system
    deleted_at   = Column(DateTime, nullable=True)   # GDPR soft delete # system


class Case(Base):
    __tablename__ = "cases"

    id               = Column(String, primary_key=True, default=gen_uuid)
    user_id          = Column(String, ForeignKey("users.id"), nullable=True)
    org_id           = Column(String, ForeignKey("orgs.id"), nullable=True)
    title            = Column(String)              # AI_POPULATED initially
    victim_context   = Column(Text)                # USER INPUT
    overall_severity = Column(String, default="none")  # AI_POPULATED
    status           = Column(String, default="open")  # USER INPUT

    user     = relationship("User", back_populates="cases")
    evidence_items = relationship("EvidenceItem", back_populates="case")


class EvidenceItem(Base):
    __tablename__ = "evidence_items"

    id                   = Column(String, primary_key=True, default=gen_uuid)
    case_id              = Column(String, ForeignKey("cases.id"), nullable=False)
    raw_content          = Column(Text, nullable=False)    # USER INPUT
    content_hash         = Column(String)                  # system · SHA-256
    hash_chain_previous  = Column(String)                  # system · chain link
    platform             = Column(String)                  # AI · detected from URL
    timestamp_utc        = Column(DateTime, default=datetime.utcnow)  # system

    case           = relationship("Case", back_populates="evidence_items")
    classification = relationship("Classification", back_populates="evidence_item",
                                  uselist=False)  # 1-to-1


class Classification(Base):
    __tablename__ = "classifications"

    id               = Column(String, primary_key=True, default=gen_uuid)
    evidence_item_id = Column(String, ForeignKey("evidence_items.id"),
                              nullable=False, unique=True)
    severity         = Column(String, default="none")       # AI
    confidence       = Column(Float,  default=0.0)          # AI
    summary          = Column(Text)                         # AI · English
    summary_de       = Column(Text)                         # AI · German
    potential_consequences    = Column(Text)                # AI
    potential_consequences_de = Column(Text)                # AI

    categories = relationship("Category", secondary=classification_categories)
    laws       = relationship("Law",      secondary=classification_laws)
Say out loud "The user provides email and content. The system generates IDs, SHA-256 hashes, UTC timestamps, and auto-detects platform from URL. The AI populates severity, categories, laws, summaries, and consequences. Each column knows where it came from."

The flow direction is users → cases → evidence_items → classifications because a case is the unit of legal documentation — a harassment incident is one case with many evidence items, not orphaned evidence sorted afterwards.

action item 2

User CRUD — seven endpoints, magic link only.

Zisis asked: show me the user-management endpoints.

Magic link only. Three reasons, in this order:

  • Victims under stress forget passwords.
  • No password DB = no breach risk.
  • One-time 15-min tokens are phishing-resistant.
backend/app/routers/auth.py 
router = APIRouter(prefix="/auth", tags=["auth"])


def _require_user(authorization: str | None):
    """Extract and validate session token from the Authorization header."""
    if not authorization:
        raise HTTPException(status_code=401, detail="Authorization header required")
    token = authorization.replace("Bearer ", "").strip()
    user = get_user_by_session(token)
    if not user:
        raise HTTPException(status_code=401, detail="Invalid or expired session")
    return user, token

action item 3

Case CRUD — cases are created implicitly.

Zisis asked: how does the user create a case?

They don't. The user never sends a POST /cases. They paste content to POST /analyze/ingest, and the system creates both the evidence row and the case row in one call. The product mental-model matches the user's: they think "I want to document what happened," not "I should create a case object."

backend/app/routers/analyze.py 
@router.post("/ingest")
def ingest_content(req: IngestRequest, db: Session = Depends(get_db)):
    """
    Classify text and optionally persist to a case.
    If case_id is provided, evidence is saved to the database.
    Otherwise, returns ephemeral result (backward compatible).
    """
    text = req.text
    if not text.strip():
        raise HTTPException(status_code=400, detail="Text content is required")

    # 1. Classify (pure AI call — no DB yet)
    try:
        classification = classify(
            text,
            victim_context=req.victim_context,
            jurisdiction=req.jurisdiction,
            user_lang=req.user_lang,
        )
    except ClassifierUnavailableError as e:
        raise HTTPException(status_code=503, detail=str(e))

    # 2. System-generated fields
    content_hash = hash_content(text)
    captured_at  = capture_timestamp()

    # 3. Persist if a case_id was given (and create the case if it doesn't exist)
    if req.case_id:
        case = db.query(DBCase).filter_by(id=req.case_id).first()
        if not case:
            raise HTTPException(status_code=404, detail="Case not found")

        previous_hash = get_last_hash(db, req.case_id)
        evidence = add_evidence_with_classification(
            db=db, case_id=req.case_id, text=text,
            classification_result=classification,
            content_type="text",
            previous_hash=previous_hash,    # SHA-256 chain
            classifier_tier=1,
        )
        return {"evidence_id": evidence.id, "case_id": req.case_id, ...}

    # 4. Ephemeral path — return an EvidenceItem without saving
    return {"evidence": EvidenceItem(...), "persisted": False}
Say out loud "Cases are created implicitly. The user pastes content via POST /analyze/ingest — the system classifies, hashes, timestamps, chains, and persists — all in one call. Read, export, and PDF follow standard REST afterwards."
action item 4 · !!!

AI flow — input, context, system prompt.

Zisis asked (with three exclamation marks): design the AI flow. What is the input? What is the context? What is the system prompt?

Input is always a string. It arrives three ways — paste, URL scrape, screenshot OCR — but by the time classify() is called the input is text. One format, one classification path.

Context lives entirely in the system prompt. It tells the model who it is, what to analyse, which enums it's allowed to produce, how to handle obfuscation, and how severity is defined. No RAG at classification time; no DB lookup; no external context. System prompt + user text is the whole input.

backend/app/services/classifier_llm_v2.py 
SYSTEM_PROMPT = """Du bist SafeVoice — ein juristischer Klassifikator für digitale Gewalt in Deutschland.

Du analysierst Texte aus sozialen Medien (Kommentare, DMs, Posts) und klassifizierst
sie nach deutschem Strafrecht.

WICHTIG:
- Verstehe Tippfehler, Slang, absichtliche Verschleierung (z.B. "f0tze", "stirbt" statt "stirb")
- Wenn unklar: im Zweifel FÜR das Opfer entscheiden (höhere Severity)
- Eine Drohung ist eine Drohung, auch wenn sie indirekt formuliert ist
- Beachte den Gesamtkontext, nicht einzelne Wörter

Gib mindestens eine Kategorie an (im Zweifel: harassment).
NetzDG § 3 gilt IMMER bei Social Media Inhalten — füge es zu applicable_laws hinzu.

SEVERITY:
- low: Grenzwertig, Verstoß gegen Nutzungsbedingungen möglich
- medium: Wahrscheinlicher Rechtsverstoß
- high:  Klarer Rechtsverstoß, Anzeige empfohlen
- critical: Schwere Straftat, sofortige Anzeige + Beweissicherung"""

Why this prompt works:

Role assignment

"Du bist SafeVoice — ein juristischer Klassifikator." Not a chatbot. The model adopts the role for the whole call.

Behavioural rule

"Im Zweifel FÜR das Opfer" — prefers false positives over false negatives. For victim-support this is correct.

Invariant

"NetzDG § 3 gilt IMMER" — guarantees we never forget the platform-obligation law.

Scale definitions

low / medium / high / critical are defined concretely, so the scale stays consistent across calls.

The user-role message is then just:

user_message = f"Klassifiziere diesen Inhalt:\n\n{text}"
Say out loud "Input is always a string. Context is the whole system prompt — role, category enum, law enum, severity definitions, the rule 'err on the side of the victim.' User message is 'Klassifiziere diesen Inhalt:\\n\\n{text}'. One turn. No RAG here."

Classification endpoint — four decisions, one call.

Zisis asked: design your classification endpoint.

backend/app/services/classifier_llm_v2.py · line ~171 
completion = client.chat.completions.parse(
    model="gpt-4o-mini",              # 15× cheaper than gpt-4o, ~90% accuracy
    temperature=0,                    # deterministic — legal must be reproducible
    max_tokens=1024,                  # enough for the JSON, not wasteful
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user",   "content": f"Klassifiziere diesen Inhalt:\n\n{text}"},
    ],
    response_format=LLMClassification,  # Pydantic class, server-enforced schema
)

The four decisions encoded in this single call:

  • gpt-4o-mini over gpt-4o — about fifteen times cheaper, roughly ninety percent of accuracy on classification tasks. Classification isn't a reasoning task where the extra capacity pays off.
  • temperature=0 — legal classification must be deterministic. Same content today and next month must produce the same result, or court-admissibility breaks.
  • .parse() instead of .create() — the modern Structured Outputs API. OpenAI validates the schema server-side and refuses invalid responses before we see them.
  • response_format=LLMClassification — a Pydantic class, not a JSON Schema string. Typed, enum-validated, and portable across model providers.
Say out loud "gpt-4o-mini, temperature zero, .parse() with a Pydantic response_format. Four decisions, one call — cheap, deterministic, schema-enforced."
action item 6

Parsing the structured output — two checks, no regex.

Zisis pointed at the OpenAI Structured Outputs docs — he wanted to see typed parsing, not manual JSON.

The old approach (still in classifier_llm.py for comparison) put a JSON schema inside the prompt and then json.loads() the raw string — defensive against markdown fences, enum-invention, and missing fields. The new approach does none of that.

backend/app/services/classifier_llm_v2.py · Pydantic schema 
class LLMClassification(BaseModel):
    """Pydantic model for the classifier's structured output.
    Schema-enforced by OpenAI server-side."""
    model_config = ConfigDict(extra="forbid")    # reject any unexpected field

    severity:                   LLMSeverity                                  # enum
    categories:                 list[LLMCategory] = Field(..., min_length=1) # enum list
    confidence:                 float             = Field(..., ge=0.0, le=1.0)
    requires_immediate_action:  bool
    summary:                    str
    summary_de:                 str
    applicable_laws:            list[LLMLaw]   # enum list (§ 185, § 241, NetzDG § 3…)
    potential_consequences:     str
    potential_consequences_de:  str

After the .parse() call, the parsing code is exactly this:

msg = completion.choices[0].message

# Check 1: did OpenAI refuse on safety grounds?
if msg.refusal:
    return None                         # → orchestrator returns 503

# Check 2: did the model produce a schema-valid result?
llm_result = msg.parsed                 # typed LLMClassification or None
if llm_result is None:
    return None                         # → orchestrator returns 503

# Map the typed LLM enums into domain objects (Severity, Category, Law)
return _to_domain(llm_result)

What the schema buys you, concretely:

  • Server-side enforcement — OpenAI refuses malformed outputs before they reach Python.
  • Type safety — msg.parsed is a validated LLMClassification instance, or None. No dict[str, Any].
  • Enum validation — if the model tried to return severity="extreme", the call is refused. Only the four values in LLMSeverity are valid.
  • Refusal handling — msg.refusal is set when OpenAI declines on safety grounds; we handle it explicitly.
Say out loud "Two checks: msg.refusal for safety declines, and msg.parsed is None for schema failure. Otherwise I get a validated Pydantic instance. No json.loads, no markdown-fence stripping, no defensive try/except. Either it's valid or it's None."

RAG — two AI layers, one uses it.

Zisis wants to know if you use RAG. Yes, but only in the second layer. The classifier above is pure prompt-in / structured-out. The case-level analysis is where retrieval earns its place.

backend/app/services/legal_ai.py 
def analyze_case_legally(case: Case) -> dict | None:
    """Deep legal analysis of an entire case using gpt-4o-mini.
    Returns a CaseAnalysis-shaped dict or a rule-based fallback."""
    if not is_available():
        return _fallback_analysis(case)

    client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])

    # ── RETRIEVE ─── pull every evidence item + its classification
    evidence_summary = []
    for ev in case.evidence_items:
        entry = {
            "author":       ev.author_username,
            "platform":     ev.platform,
            "content":      ev.content_text[:500],
            "captured_at":  str(ev.captured_at),
        }
        if ev.classification:
            entry["severity"]   = ev.classification.severity.value
            entry["categories"] = [c.value for c in ev.classification.categories]
            entry["laws"]       = [l.paragraph for l in ev.classification.applicable_laws]
        evidence_summary.append(entry)

    # ── AUGMENT ─── structure as a context block inside the user prompt
    user_prompt = f"""Analyze this digital harassment case.

Case ID: {case.id}
Title: {case.title}
Overall severity: {case.overall_severity.value}
Victim context: {case.victim_context or 'Not provided'}

Evidence ({len(case.evidence_items)} items):
{json.dumps(evidence_summary, indent=2, ensure_ascii=False)}

Produce a structured legal assessment. Use German primary for _de fields
and English secondary for _en fields."""

    # ── GENERATE ─── same Structured Outputs pattern as the classifier
    completion = client.chat.completions.parse(
        model="gpt-4o-mini",
        temperature=0.2,
        max_tokens=2048,
        messages=[
            {"role": "system", "content": LEGAL_SYSTEM_PROMPT},
            {"role": "user",   "content": user_prompt},
        ],
        response_format=CaseAnalysis,       # strategy, charges, actions, risk, gaps
    )

    msg = completion.choices[0].message
    if msg.refusal or msg.parsed is None:
        return _fallback_analysis(case)
    return msg.parsed.model_dump()

That's textbook retrieve → augment → generate. The classifier above is single-turn, single-input, no retrieval. The legal layer above is retrieval across N evidence items so the model can reason about patterns, escalation, and cross-references.

Say out loud "The classifier is prompt-in, structured-out — no RAG. The legal-analysis layer uses RAG: it retrieves all the case's evidence plus classifications from the database, structures them as a context block, and sends to the same gpt-4o-mini with a different Pydantic schema — CaseAnalysis — for strategy, strongest charges, recommended actions, risk assessment, and evidence gaps. One model family, two jobs."
