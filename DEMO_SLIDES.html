"Six tables, one story. We remember a person, freeze the evidence, group it into a case, record the AI's verdict separately, and anchor that verdict to a shared vocabulary of categories and German law paragraphs."
action item 1 of 6

Every column, labelled.

TaskIdentify DB table attributes as input_from_the_user or populated_by_AI. 
class Classification(Base):
    __tablename__ = "classifications"
    id               = Column(String, primary_key=True, default=gen_uuid)    # system_generated
    evidence_item_id = Column(String, ForeignKey("evidence_items.id"), unique=True)  # 1-to-1
    severity         = Column(String, default="none")    # AI_POPULATED
    confidence       = Column(Float,  default=0.0)       # AI_POPULATED
    summary          = Column(Text)                      # AI_POPULATED · en
    summary_de       = Column(Text)                      # AI_POPULATED · de
    classified_at    = Column(DateTime, default=datetime.utcnow)  # system_generated

design decisionThree sources of truth — user input, AI populated, system generated — never mixed. The label lives as an inline comment in backend/app/database.py, mirrored in schema.dbml. Anyone reading either file can answer "who wrote this column?" in one glance.

"Every column in every table is labelled. User input, AI populated, or system generated. That's the mental model I use when reading or writing any query."
action item 2 of 6

User CRUD — eight endpoints, zero passwords.

TaskDesign the endpoints for creating and updating a user.
POST /auth/login Request magic link · creates user on first login POST /auth/verify Exchange token for session GET /auth/me Read — current user PUT /auth/me Update — display_name, language POST /auth/logout Invalidate session DELETE /auth/me Soft delete — GDPR Art. 17 DELETE /auth/me/emergency Hard delete — purges all cases + evidence GET /auth/me/export Art. 20 GDPR data export (JSON)

design decisionNo passwords, ever. A victim under stress forgets passwords; a password database we don't have cannot leak. Magic link → 15-minute token, single use, 256 bits of entropy. Sessions expire, tokens chain to sessions. backend/app/routers/auth.py.

"Eight endpoints in auth.py. Magic-link only — no passwords means no password breach. Update is PUT /auth/me, delete is a two-level system: soft-delete for GDPR Art. 17, hard-delete for emergency erasure."
action item 3 of 6

Case CRUD — six endpoints, implicit creation.

TaskDesign the endpoints for filling, updating, deleting a case.
GET /cases/ List — all cases for current user GET /cases/{id} Read — case + evidence + classifications POST /cases/ Create — rarely used directly PUT /cases/{id} Update — title, status, notes DELETE /cases/{id} Delete — cascades to evidence + classifications POST /cases/{id}/evidence Filling — attach more evidence to existing case

design decisionIn practice, cases are created implicitly by POST /analyze/ingest — the user pastes content and the backend creates evidence + classification + case in one transaction. POST /cases/ exists for completeness but the real flow never touches it. Let the system do the bookkeeping; the victim has enough to do.

"Six CRUD endpoints in cases.py, but the user rarely sees them. The real entry point is POST /analyze/ingest — the user pastes content, and evidence + classification + case are created in a single transaction. Zero clicks between hurt and help."
action item 4 of 6 · !!!

AI flow — input, context, system prompt.

TaskDesign the AI flow. What is the input? What is the context? What is the system prompt?

Input

Always a string. Arrives three ways — paste, URL scrape, or screenshot OCR — but by the time classify() runs, it's plain text. One format, one path.

Context

Lives entirely in the system prompt. No RAG here — the context is stable rules + four worked examples (few-shot), not retrieved documents. RAG shows up at the case level — last slide.

action item 5 of 6

Classification — four entry points, one brain.

TaskDesign your classification endpoint / endpoints.
POST /analyze/text Classify one string — fastest path, no DB writes POST /analyze/ingest Create evidence + classify + create case — the workhorse POST /analyze/url Scrape URL → classify the scraped text POST /analyze/case Case-level analysis across N evidence items (RAG)

design decisionFour endpoints, one brain: every route funnels into classifier_llm_v2.classify_with_llm(). The difference is only where the string comes from and what we do with the result. backend/app/routers/analyze.py.

"Four endpoints in analyze.py, but they all call the same classify_with_llm() function. Separating where the text comes from from what we do with the verdict means the classifier itself stays stateless and unit-testable."
action item 6 of 6

.parse() · schema-enforced, one call.

TaskParse the output using OpenAI Structured Outputs. 
completion = client.chat.completions.parse(
    model="gpt-4o-mini",
    temperature=0,
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user",   "content": f"Klassifiziere diesen Inhalt:\n\n{text}"},
    ],
    response_format=LLMClassification,   # ← Pydantic, server-enforced
)

msg = completion.choices[0].message
if msg.refusal: return None     # OpenAI declined on safety
parsed = msg.parsed             # typed Pydantic instance, or None
if parsed is None: return None  # schema failed → clean 503
return _to_domain(parsed)

design decisionPydantic schema with LLMCategory and LLMLaw as enums — the model cannot invent a category or a paragraph. Refusal is explicit (not exception), and None propagates cleanly to a 503. No json.loads, no regex cleanup. classifier_llm_v2.py:227.

"One call. The response is either a typed Pydantic instance with schema-enforced enums, or it's None — and None means we return a clean 503. No JSON parsing, no malformed-output retries."
bonus · RAG

Two AI layers. Only one uses RAG.

LayerJobRAG?Model
Classifier One piece of content → severity, categories, laws No gpt-4o-mini
Legal AI Whole case → strategy, precedents, cross-references Yes gpt-4o-mini

services/legal_ai.py retrieves all evidence + classifications from a case, structures them as a context block, sends to OpenAI. Classic retrieve → augment → generate. The Week 9–10 deliverable.

"Classification doesn't need RAG — it's one string in, one verdict out. RAG earns its place at the case level, where the model has to reason across N pieces of evidence at once."

SQLAlchemy ORM + Alembic migrations. Categories and laws are seeded on startup so a fresh database ships with reference data — no "empty dropdown" state on first launch.
