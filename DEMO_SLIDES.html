60-second opener

SafeVoice helps victims of digital harassment document an incident in 30 seconds and export a court-ready report.

Since our last meeting I've implemented the six action items — real SQLAlchemy database, user endpoints with magic-link auth, case endpoints, the AI flow fully documented, structured outputs via Pydantic, and I removed the 3-tier classifier fallback.

Today I want to walk you mostly through the AI flow — the three exclamation marks you put on that one.

Action items 1 – 3 · structure

Real database. Magic link. Implicit cases.

Database

SQLAlchemy ORM + Alembic migrations. SQLite locally, Postgres on Railway. Eight tables + multi-tenancy (orgs, org_members) added April 12.

User input → email, language, content, victim_context.
System-generated → IDs, SHA-256 hashes, UTC timestamps.
AI-populated → severity, categories, laws, summaries.

Auth + endpoints

Seven auth endpoints, magic link only. No passwords — victims under stress forget passwords, no password DB means no breach risk, tokens are 15-min phishing-resistant.

Cases are created implicitly. The user pastes content via POST /analyze/ingest — the system creates both evidence and case in one call. They never think "I'll create a case."

action item 1 · database schema

Eight tables. Three sources of truth.

Plus multi-tenancy — orgs and org_members (added 12 April) carry NGO case-workers and API keys. SQLAlchemy ORM + Alembic migrations. Categories and laws are seeded on startup so a fresh database ships with reference data.

AI flow · input + context

One string in. One system prompt.

Input

Always a string. It arrives three ways — paste, URL scrape, or OCR — but by the time classify() is called, the input is text. One format, one path.

Context

Lives entirely in the system prompt. Role, category enum, law enum, severity definitions, and the core rule "err on the side of the victim" — prefer false positives to false negatives.

WICHTIG:
- Verstehe Tippfehler, Slang, absichtliche Verschleierung
- Wenn unklar: im Zweifel FÜR das Opfer entscheiden
- NetzDG § 3 gilt IMMER bei Social Media Inhalten

SEVERITY:
- low / medium / high / critical (defined explicitly)
AI flow · classification + parse

.parse() · schema-enforced · one call.

completion = client.chat.completions.parse(
    model="gpt-4o-mini",          # 15× cheaper, ~90% accuracy
    temperature=0,                 # deterministic — legal must be
    max_tokens=1024,
    messages=[
        {"role": "system", "content": SYSTEM_PROMPT},
        {"role": "user", "content": f"Klassifiziere diesen Inhalt:\n\n{text}"},
    ],
    response_format=LLMClassification,   # Pydantic, server-enforced
)

msg = completion.choices[0].message
if msg.refusal: return None       # OpenAI declined on safety
parsed = msg.parsed               # typed instance or None
if parsed is None: return None    # schema-conformance failed
return _to_domain(parsed)

No json.loads. No code-fence stripping. No defensive try/except around malformed JSON. Either I get a validated Pydantic instance, or I get None and return a clean 503.

bonus · RAG

Two AI layers. One uses RAG.

LayerJobRAG?Model
Classifier One piece of content → severity, categories, laws No gpt-4o-mini
Legal AI Whole case → strategy, precedents, cross-references Yes gpt-4o-mini

services/legal_ai.py retrieves all evidence + classifications from a case, structures them as a context block, sends to OpenAI. Classic retrieve → augment → generate. The Week 9–10 deliverable.

The split is deliberate: classification must be deterministic and single-input. Case-level reasoning is retrieval across N evidence items — that's where retrieval earns its place.
